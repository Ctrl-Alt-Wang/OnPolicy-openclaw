# Medical Research Agent - 医学研究调研助手

## 概述

这个技能帮助用户对医学主题进行深度前期调研。它通过三个阶段工作——规划、检索、汇总——生成一份 Markdown 调研报告(中间文件)，最终交付默认都应是由 Markdown 调研报告转换的 Word 报告（`.docx`）和 PDF 报告（`.pdf`），而不是只停留在对话文本或 TXT/Markdown 中间文件。

**⚠️ 对话输出约定：完整报告正文只写入文件，禁止在对话中全文输出。对话中仅输出精华摘要（见第四阶段末尾规范），避免长文截断或卡死。**

三阶段设计的原因：医学文献调研不是简单的关键词搜索。好的调研需要先想清楚"从哪些角度去看"（规划），然后系统地收集证据（检索），最后综合分析得出洞察（汇总）。这三步确保调研有深度、有广度、有结构。

## ⚠️ 调用方式（重要）

本技能依赖 `scripts/search.py` 进行文献检索，它封装了 Infox-Med 医学文献 API。

**必须通过命令行参数传入关键词**，不要直接运行脚本（直接运行会报错缺少必要参数）。

### 分步操作

#### 步骤 1：从用户问题中提取搜索关键词

分析用户的研究问题，根据检索计划中的维度提取 3-5 个精准搜索关键词。关键词应短小精准，可直接喂给搜索工具。

#### 步骤 2：通过命令行调用搜索脚本

使用 `run_command` 工具执行以下命令，**将关键词替换为从用户问题中提取的实际值**：

```bash
python3 <skill_directory>/scripts/search.py --keywords "关键词1" "关键词2" "关键词3" --output "<输出文件路径>" --summary-limit 8
```

**完整示例：**

```bash
python3 <skill_directory>/scripts/search.py --keywords "colorectal cancer organoid" "drug resistance mechanism" "chemotherapy tolerance" --output "/tmp/search_results.json" --summary-limit 8
```

```bash
python3 <skill_directory>/scripts/search.py --keywords "肠道微生物" "PD-1免疫治疗" "响应性" --output "/tmp/search_results.json" --summary-limit 8
```

**参数说明：**

| 参数 | 必选 | 说明 |
|------|------|------|
| `--keywords` | ✅ 是 | 1 个或多个搜索关键词，每个用引号包裹 |
| `--output` | ❌ 否 | 结果输出文件路径，不指定则输出到终端 |
| `--summary-limit` | ❌ 否 | 使用 `--output` 时，终端摘要最多展示的文献条数（默认 8） |

> **注意**：每个关键词并行请求 API（最多 4 个同时），每个最多返回 10 篇文献（近 10 年），结果按文献 ID 自动去重。建议使用 `--output` 参数将结果保存到文件，避免终端输出过长被截断。使用 `--output` 时终端会自动打印按影响因子排序的摘要；先用摘要规划分析方向，除非进入最终引用核验，不要读取完整 JSON。

#### 步骤 3：读取并展示结果

如果使用了 `--output` 参数，用 `view_file` 读取输出文件；否则直接从命令输出中获取结果。

将结果整理后用于后续的维度分析和报告生成。

**返回字段说明：**

| 字段 | 说明 |
|------|------|
| `id` | 文献唯一 ID |
| `title` | 文献标题 |
| `pmid` | PubMed ID |
| `abstract` | 摘要（英文或中文） |
| `url` | Infox-Med 详情页链接 |
| `publish_time` | 发布时间（YYYY-MM 格式） |
| `docIf` | 期刊影响因子 |
| `_matched_keywords` | 命中的关键词列表 |

更多细节参见 `references/search_tools.md`。

## 核心流程

### 第一阶段：制定检索计划

将用户的研究问题拆解为 4-6 个搜索维度。每个维度代表一个不同的观察视角，这样做是为了避免只看到"主流结论"而遗漏矛盾发现和空白区域。

推荐的维度模板（根据具体问题灵活调整）：
1. **主流共识与权威结论** — 建立基线认知，后续才能识别"非共识"
2. **关键机理与方法学证据** — 理解核心机制和研究方法
3. **主题演变与里程碑** — 构建近十年时间脉络，把握趋势
4. **非共识/反例与相反证据** — 刻意寻找与主流不同的声音
5. **临床/应用与转化挑战** — 聚焦真实世界落地难点

为每个维度设计 3-5 个精准、简短的搜索查询词（可直接喂给搜索工具）。输出结构如下：

```json
{
  "plan": {
    "goal": "可检索的目标描述",
    "domain": "研究领域",
    "dimensions": [
      {
        "id": "d1",
        "name": "维度名称",
        "rationale": "为什么需要这个维度",
        "queries": ["查询1", "查询2", "查询3"]
      }
    ]
  }
}
```

### 第二阶段：执行文献检索

对每个维度的每个查询词调用 `search()` 函数。为了提高效率，尽量并行处理多个维度。

**文献筛选优先级**（从高到低）：
- 综述/系统评价/RCT/大样本真实世界证据
- 高影响因子期刊的机理研究
- 方法学与基准研究
- 权威机构白皮书

每个维度保留 10 篇以上高质量文献。近 5 年优先，但可追溯更早的里程碑文献。

对每个维度，整理出：
- `useful_docs`：筛选后的文献列表（含 title、pmid、year、docIf、abstract、url、why_useful）
- `summary_points`：该维度关键信息要点（最多 5 条）
- `notes`：不足或需要补充的说明

### 第三阶段：生成调研报告

基于检索结果生成结构化 Markdown 报告。报告定位为**决策手册**而非分析综述，每个章节都必须回答"所以呢？"提供可落地的行动指引。

**⚠️ 重要：完整报告正文必须写入语义化命名的 Markdown 文件。根据报告主题生成文件名，格式为 `{主题描述}_医学研究空白点调研.md`（如 `肿瘤免疫耐药_医学研究空白点调研.md`、`肠道菌群与PD1响应_医学研究空白点调研.md`）。从报告标题中提取核心关键词命名，确保不同主题的报告不会互相覆盖。严禁使用固定的 `report.md`。**

报告模板和详细示例参见 `references/output_template.md`。

### 第四阶段：导出交付件

只要进入本技能并完成了一份正式调研报告，必须把最终报告写入文件并导出交付件。是否导出文件不以用户是否说出"导出报告"为前提；默认就是要导出。

**强制完成门槛：**

- 当用户说"写一份报告""做个报告""帮我写报告""整理成报告"这类请求时，Markdown 文件只算中间产物，不算最终交付
- 这类任务只有在 .md、.docx、.pdf 都已处理完成后，才允许结束当前回复
- 不允许因为"正文已经写好"就提前收尾
- 不允许把"我先给你一版 Markdown/TXT"当成默认完成状态
- 如果某一导出失败，必须继续完成剩余可导出的格式，并明确写出"当前未完成项"而不是宣称任务已完成

默认导出顺序：

1. 先将 Markdown 正文保存为语义化命名的文件（如 `肿瘤免疫耐药_医学研究空白点调研.md`）
2. 再调用导出脚本生成对应的 .docx 和 .pdf

命令示例（⚠️ 文件名必须唯一，禁止使用固定的 `report.md`）：

```bash
python <skill_directory>/scripts/export_report.py --input "主题描述_医学研究空白点调研.md" --format all
```

要求：

- 只要生成了正式报告，默认必须同时生成 `.docx` 和 `.pdf`
- 即使用户没有提"导出""保存""生成文件"，也要自动导出这两个文件
- 只有用户明确说"不要文件""只在聊天里给我文本"时，才可以不生成文件
- 不得只返回 TXT，亦不得仅保留 Markdown 中间文件
- 对"写报告"类请求，禁止把 `.md` 作为唯一附件或唯一文件链接返回给用户
- 如果导出失败，必须明确说明是 `pandoc`、浏览器或写权限哪个环节失败
- Word 导出优先使用 `pandoc`；若本机缺少 `pandoc`，应该自行尝试安装 `pandoc`，如果失败则自动切换到 `scripts/export_report.py` 的内置 `.docx` 兜底生成方案，而不是直接放弃 Word 导出
- 如果 `.docx` 导出失败但 `.pdf` 仍可导出，则必须继续完成 PDF 导出，不能因为 Word 失败而中断整个交付
- 如果 `.pdf` 导出失败但 `.docx` 可导出，则仍需保留 Word 交付件，并只汇报 PDF 的失败原因
- 导出 PDF 时，必须使用 `scripts/export_report.py` 内置的蓝白医学报告模板样式，不得使用浏览器默认无样式打印结果
- 如果 PDF 导出环境为 `root + Chromium/Chrome`，必须自动追加 `--no-sandbox` 启动参数，而不是把它作为未解决问题抛回给用户
- 收尾时默认汇报"已生成哪些文件 + 哪些文件失败 + 失败原因"，不要给用户罗列下一步选项菜单，除非用户明确要求你提供方案选择
- 非致命环境告警（如 Chromium 的 `dbus` stderr 噪音）不应作为结果重点回报；只有真正导致导出失败时才需要提及
- 除非用户明确要求 `git commit`、`git add` 或提交版本控制，不要主动尝试任何 git 提交动作，也不要在结果中汇报 git 提交失败

**⚠️ 对话精华摘要（文件导出完成后，在对话中仅输出以下结构，不输出其他正文内容）：**

```
## 📋 调研完成：《报告标题》

### 核心研究空白点
- **G1 空白点名称**：根因一句话 + 解决方向一句话
- **G2 空白点名称**：……
- **G3 空白点名称**：……
（共列出全部 G1-G7）

### 高价值课题推荐
- **课题1：课题名称** | 影响等级 ★★★★★ | 适合：[团队类型]
  核心假说一句话 + 最大风险一句话
- **课题2：……**
- **课题3：……**
（共列出全部课题）

### 优先立项建议
3-5 句话给出落地性建议。

📎 已生成文件：{主题描述}_医学研究空白点调研.docx | {主题描述}_医学研究空白点调研.pdf（列出实际文件名）
```

报告的核心结构：

```
# 《主题》研究空白点深度调研报告

# 核心洞察速览
（4 项卡片：核心演变趋势 / 核心科学瓶颈 / 最具价值研究方向 / 最高优先级行动）

# 一、引言与分析范围
（2-3 句话界定主题边界、对象、时间窗口、调研目的）

# 二、背景和意义
（详细阐述该领域的临床痛点、科学价值、当前研究的必要性及潜在的社会与经济效益，字数约 1000 字左右）

# 三、研究主题与主要研究方向解析
## 3.1 主题演变概览（包含时间段、主要研究主题、代表性研究摘要与来源标号、驱动因素、挖掘度）
## 3.2 演变趋势解析与非共识路径识别（寻找与主流不一致的结论并简述核心机制/推论）
## 3.3 领域演进时间轴图谱（使用高级美观的 Mermaid timeline 表达）

# 四、领域现状与决策基线
## 4.1 当前共识与关键证据（3-5条，每条附 why it matters）
## 4.2 技术与方法学演变趋势（聚焦对决策有影响的趋势，表格形式列出世代/范式/成熟度/启示）
## 4.3 现存分歧与非共识路径（至少 3-4条，每条含机制+推论+启示）

# 五、空白点全景图与可执行映射
（强烈警告：由于维度较多，【每个空白点必须单独输出一个独立的表格卡片】，严禁将所有空白点合并横向展开为一个大表！每个空白点标题格式为 `**空白点 G1：空白点名称**`。每个表格表头为【维度】与【描述剖析】。输出 5-7 个核心研究空白点。表格必须包含以下 7 个内部维度：根因分析 / 临床影响 / 解决路径 / 推荐研究模式 / 适合团队 / 技术可行性 / 条件适配度）

# 六、高价值研究课题提案
（强烈警告：【每个课题必须先输出一个独立标题，再输出一个独立表格】，严禁将多个课题合并横向展开为一个大表！标题格式统一为 `**课题 N：课题名称**`，作为图示化标题行单独占一行。每个独立课题采用标准 Markdown 表格呈现，表头分别为【维度】与【描述】。输出 3-5 个课题。课题必须包含如下十个维度名：方向 / 空白点映射 / 核心假说 / 研究路径 / 推荐基金方向 / 推荐期刊 / 主要风险 / 影响等级 / 成本预估 / 预期产出。
极度重要：在每个课题表格下方，必须新增名为"**工程化落地执行方案：**"的独立文本段落，深度描述每个阶段应当做什么、有何风险控制、需要什么资源支撑、阶段成功的标志为何，如果失败了应如何复盘应对。）

# 七、研究优先级与立项建议
## 7.1 定向决策指南（依据已有信息给出导向性建议：哪个适合药企转化？发文章？资源有限团队等）
## 7.2 推荐策略优先级矩阵（产出推荐策略表，含立项目标、优先级星级、适配课题与风险）

# 八、决策里程碑与分阶段规划
## 8.1 关键决策节点（阶段门设计，含Go/No-Go条件与备选方案）
## 8.2 风险控制与备选方案（技术/样本/数据/时间四类结构化风控图表）

# 九、参考文献
（强制要求使用数字排序编号。参考文献采用"作者. 标题. 期刊. 年份"格式，必须隐藏期刊的 IF 值。请必须提供 20 个以上高质量参考来源。）
```

标题规范要求：

- Markdown 正文最顶部必须显式写出正式报告标题，并直接使用真实标题行，推荐格式为 `# 肿瘤免疫治疗近三年研究空白点深度调研报告`
- 标题必须根据用户问题自动生成，贴合研究对象、疾病范围、时间窗口和任务类型，不能机械照抄模板占位符
- 除非用户明确指定标题，否则应由模型自主生成一个自然、正式、可交付的报告标题
- 禁止把 `标题：` 这三个字作为最终标题的一部分输出给用户；`标题：...` 只可作为内部兼容写法，不推荐作为最终正文格式
- 禁止让正文直接从 `# 核心洞察速览` 开始
- 如果缺少这个标题行，导出的封面标题会被误判为正文第一节标题

**写作要求**：
- **决策导向**：避免纯分析性描述，每个要点必须给出"这意味着什么"或"应该怎么做"
- **背景撰写**：第二部分的"背景和意义"必须字数足够详实，深入探寻背景价值（1000字左右），提升研究的学术沉淀感
- **去冗余**：同一课题/空白点集中在一处讨论，不拆分到多章节
- **颗粒度**：建议具体到"做什么实验→用什么技术→需要什么样本"级别
- **风险量化**：每个建议方向附带可行性/风险等级和成本预估
- **规范引用**：禁止正文内容中出现任意形式的数字引用，参考文献除外。
- 仅输出 Markdown 正文，不输出 JSON 或解释性旁白
- 证据不足时标注"（证据不足，需继续检索）"
- 第五章强烈警告：禁止使用单一大表、段落卡片、项目符号或编号列表；【必须为每个空白点单独输出 1 个完整的 Markdown 表格】，共 5-7 个（G1-G7），每个表格采用两列格式（维度与描述剖析）。
- 第五章每个单元格都必须写实质内容，必须进行深入和充实的论述刻画，禁止留空、写"同上"或只写星级不解释
- 第五章表格内的"技术可行性"与"条件适配度"统一采用"星级 + 括号说明"格式，例如 `★★★★（单细胞+空间转录组已成熟）`
- 第七章的优先级与立项建议必须紧扣前文痛点，给出极具落地性的针对性建议。
- 只要完成正式报告，默认同时导出 `.docx` 和 `.pdf`；只有用户明确说"只要文本"时才可以不生成文件
- 第六章每个课题必须在表格前单独输出一个醒目的标题行，格式严格为 `**课题 N：课题名称**`，不得直接让表格裸露出现
