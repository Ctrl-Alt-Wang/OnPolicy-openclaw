# 国自然基金申请书撰写 skill

把一份国家自然科学基金申请书的撰写拆解为「准备 → 正文 → 修改优化」三阶段、十余个模块，
每个模块有明确的字数范围、输出格式、引用要求和关键要点。撰写过程**围绕一张 issues CSV
契约推进**：每个模块是一条带状态、依赖和验收标准的任务行，状态推进即进度推进。

## 设计前提（重要）

- **模型无关、工具无关**：流程与规格用通用方式描述，便于部署到任意足够强的大模型上，
  不绑定特定厂商的工具实现。
- **文献绝不编造**：凡需要文献的环节，统一调用文献检索工具（在 InfoX-Med 环境中即
  内部医学文献检索 skill / API），获取近 10 年真实文献。每条引用按**文献三态**
  （`pending` / `placeholder` / `verified`）如实标注，未 `verified` 不得标该任务 DONE。
  详见 `assets/citation-status-guide.md`。
- **输出形态灵活**：可一次产出整份申请书，也可只写某一模块，或只优化某一段，
  由用户诉求决定。

## 第一步：判断用户诉求

1. **写整份申请书** → 进入「契约驱动的完整流程」。
2. **只写某一/几个模块** → 直接查 `references/module-specs.md` 对应条目按规格产出；
   若依赖前序内容（如研究假设依赖研究现状），先索取或基于已有信息合理补全。
3. **修改/优化已有段落** → 进入「修改和优化」阶段，见 `references/revision-optimization.md`。

## 第二步：收集输入（MVP：灵活，不强卡字段）

最理想起始输入：**研究领域、已有成果、研究方向**；可选：已发表文献、用户上传数据、
项目类别（面上/青年/地区等）与年限。

MVP 原则：输入齐全直接开始；不全则合理推断 + 用一两个问题补齐**关键**缺口，
不一次抛出一长串必填项；始终允许「先写起来，后面再补」。

## 契约驱动的完整流程

### 1. 建立契约
从 `assets/issues-template.csv` 复制一份，按用户课题填好各行的 Title / Word_Spec / Notes。
模板已预置全部 19 条任务及其依赖关系。契约的列定义与用法见 `references/issues-csv-guide.md`。

### 2. 逐行推进
反复执行直到所有任务 DONE / SKIP：

1. 选一条 `Depends_On` 已全部满足（DONE/SKIP）的 TODO 任务，置 `DOING`。
2. 若 `Citation_Need=yes`：先调用文献检索工具拿到近 10 年真实文献，
   更新 `Citation_Status`（pending → placeholder → verified）。
3. 查 `references/module-specs.md` 对应模块条目，按其字数/格式/要点撰写。
4. **撰写后强制字数自检**：核对输出字数是否落在 `Word_Spec` 区间内；
   不足则扩写，超出则精简，达标后方可继续。
5. 满足 `Acceptance` 后置 `DONE`。

### 3. 门控规则（不可协商）
- **依赖门控**：标 DONE 前，确认 `Depends_On` 列出的任务全部 DONE/SKIP，否则不得 DONE。
- **文献门控**：`Citation_Need=yes` 的任务，`Citation_Status` 未达 `verified` 不得 DONE
  （检索工具不可用时如实保留 placeholder 并告知用户）。
- **字数门控**：输出字数未落入 `Word_Spec` 区间不得 DONE。

### 模块顺序与归属（与思维导图一致）

- **准备**：P1 选题分析 → P2 中文摘要 → P3 摘要翻译 → P4 研究基础和工作条件。
- **立项依据**：L1 研究意义 → L2 研究现状 → L3 研究假设 → L4 合并立项依据。
- **内容目标问题**：C1 研究内容 → C2 研究目标 → C3 关键科学问题。
- **方案可行性**：S1 研究方案 → S2 技术路线 → S3 可行性分析
  （可行性分析产品编号为 19，但归此大类，排技术路线之后）。
- **特色创新**：F1 特色与创新。
- **计划结果**：Y1 年度研究计划 → Y2 预期研究结果。
- **收尾**：Z1 参考文献（全文统一输出一次）、Z2 研究基础与工作条件汇置正文末尾。

> 进度 = DONE 任务数 / 总任务数，对应产品界面的「总进度」。

## 修改和优化阶段

用户选中任意已生成段落进行修改：在契约里把对应任务从 DONE 退回 DOING，
按准确性/规范性/连贯性/透明性/一致性/深度六维度打磨，仅改选中段落，
不得违反该模块原有规格，改完再标 DONE。详见 `references/revision-optimization.md`。

---

## 输出格式约定

- 默认在对话中以 Markdown 输出，结构清晰（标题、字数标注、引用编号）。
- 用户要求成稿文档时，可整合为 .docx 等格式交付（在支持文档生成的环境中）。
- 技术路线以可渲染流程图描述（如 Mermaid 或结构化步骤）输出。
- 每个模块输出末尾标注实际字数，便于核对字数门控。

## 不可协商的安全底线

- **绝不编造**文献、数据或事实；缺证据时如实标 `placeholder` 并询问用户。
- **状态如实**：绝不把 placeholder 谎标 verified，绝不把未达依赖的任务标 DONE。
- **不擅自改动**用户未选中的内容。
